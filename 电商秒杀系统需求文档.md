## 1. 项目概述

### 1.1 项目背景

电商秒杀是互联网高并发场景的典型代表。本项目旨在构建一个功能完整的秒杀系统，用于展示对高并发、分布式架构的理解，适合作为求职实习的核心项目。

### 1.2 项目目标

| 目标 | 描述 |
|------|------|
| 业务完整 | 覆盖用户注册登录、商品浏览、秒杀抢购、订单管理、模拟支付全流程 |
| 高并发处理 | 支持秒杀场景下的高并发请求，通过 Redis 缓存 + 消息队列削峰填谷 |
| 数据一致性 | 保证库存不超卖、订单不重复 |
| 面试友好 | 涵盖面试高频考点：分布式锁、缓存穿透/击穿/雪崩、消息队列、事务一致性 |

### 1.3 用户角色

| 角色 | 描述 |
|------|------|
| 普通用户 | 注册、登录、浏览商品、参与秒杀、查看订单、模拟支付 |
| 管理员 | 管理商品、创建秒杀活动、查看秒杀统计、管理用户 |

### 1.4 术语定义

| 术语 | 定义 |
|------|------|
| 秒杀活动 | 在指定时间段内，以优惠价格限量销售商品的活动 |
| SKU | Stock Keeping Unit，库存量单位，即商品的最小可售单元 |
| 秒杀订单 | 用户通过秒杀活动成功抢购后生成的订单 |
| 削峰填谷 | 通过消息队列将瞬时高并发请求转为异步处理，平滑系统负载 |

### 3.2 用户模块

#### 3.2.1 用户注册

| 字段 | 类型 | 必填 | 约束 |
|------|------|------|------|
| 用户名 | String | 是 | 4-20 字符，字母数字下划线，唯一 |
| 密码 | String | 是 | 6-20 字符，需包含字母和数字 |
| 确认密码 | String | 是 | 与密码一致 |
| 手机号 | String | 是 | 11 位手机号，唯一 |
| 验证码 | String | 是 | 6 位数字（模拟，固定值或随机生成） |

**业务规则：**
- 密码使用 BCrypt 加密存储
- 注册成功后自动登录，返回 JWT Token
- 用户名和手机号不可重复

#### 3.2.2 用户登录

| 字段 | 类型 | 必填 | 约束 |
|------|------|------|------|
| account | String | 是 | 用户名或手机号 |
| 密码 | String | 是 | - |

**业务规则：**
- 登录成功返回 JWT Token（有效期 30 分钟）和 Refresh Token（有效期 7 天）
- 连续登录失败 5 次，锁定账号 30 分钟
- Token 存储在 Redis 中，支持主动失效（登出）

#### 3.2.3 个人信息管理

- 修改昵称、头像、性别
- 修改密码（需验证旧密码）
- 查看个人信息

#### 3.2.4 收货地址管理

- 新增 / 编辑 / 删除收货地址
- 设置默认收货地址
- 每个用户最多保存 10 个地址

### 3.4 秒杀模块（核心）

#### 3.4.1 秒杀活动列表

- 展示当前进行中 / 即将开始的秒杀活动
- 每个活动显示：商品信息、秒杀价、原价、活动时间、剩余库存
- 活动状态：未开始 / 进行中 / 已结束 / 已售罄

#### 3.4.2 秒杀活动详情

- 展示秒杀商品详细信息
- 展示活动倒计时
- 展示秒杀规则说明

#### 3.4.3 秒杀抢购（核心流程）

**前置条件：**
- 用户已登录
- 活动状态为"进行中"
- 用户未重复购买该秒杀商品（每人限购 1 件）

**秒杀流程（三层防护 + 确认页）：**

```
用户点击"立即抢购"
    │
    ▼
[秒杀确认页] 选择收货地址、确认订单信息
    │ 确认提交
    ▼
[第一层] 接口限流（令牌桶算法 / Redis 限流）
    │ 通过
    ▼
[第二层] Redis 库存预扣（DECR 原子操作）
    │ 库存 > 0
    ▼
[第三层] 发送消息到 RabbitMQ（异步下单）
    │
    ▼
返回"排队中"，前端轮询查询结果
    │
    ▼
消费者从 MQ 获取消息
    │
    ▼
[数据库层] 创建订单 + 扣减数据库库存（事务保证）
    │
    ▼
返回秒杀结果（成功 / 失败）
```

**秒杀确认页说明：**
- 用户点击"立即抢购"后，先进入秒杀确认页
- 确认页展示：商品信息、秒杀价、收货地址列表（可选择/切换地址）
- 用户确认订单信息后，提交秒杀请求
- 确认页提交时携带 `activityId`、`goodsId`、`addressId`

**防刷策略：**
- **接口限流：** 使用 Guava RateLimiter 或 Redis + Lua 脚本限制单用户访问频率（每秒最多 1 次请求）
- **隐藏秒杀地址：** 秒杀接口 URL 动态生成，每次请求携带加密的路径参数，防止提前知道接口地址
- **IP 限流：** 同一 IP 每秒最多请求 5 次
- **用户限购：** 每个用户对同一秒杀活动只能购买 1 次

#### 3.4.4 秒杀结果查询

- 用户提交秒杀后，前端每 2 秒轮询查询结果
- 返回状态：排队中 / 秒杀成功 / 秒杀失败（库存不足 / 已购买）
- 秒杀成功返回订单号

### 3.5 订单模块

#### 3.5.1 订单创建

- 由秒杀模块异步触发（通过 RabbitMQ 消费者）
- 订单包含：订单号、用户ID、商品ID、秒杀价、订单状态、收货地址、创建时间
- 订单号生成规则：时间戳 + 机器标识 + 序列号（雪花算法）
- 订单状态：1-待支付 → 2-已支付 → 3-已发货 → 4-已完成 / 5-已取消 / 6-已退款

#### 3.5.2 订单超时自动取消

- 订单创建后 **15 分钟** 未支付，自动取消
- 使用 RabbitMQ 延迟队列（死信队列 + TTL）实现
- 取消订单时回滚库存（数据库库存 + Redis 库存）

#### 3.5.3 订单列表

- 支持分页查询当前用户的订单
- 支持按状态筛选：全部 / 1-待支付 / 2-已支付 / 3-已发货 / 4-已完成 / 5-已取消 / 6-已退款

#### 3.5.4 订单详情

- 查看订单完整信息：商品信息、收货地址、支付状态、物流信息

#### 3.5.5 订单取消

- 仅"待支付"状态的订单可手动取消
- 取消后回滚库存

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 目标值 |
|------|--------|
| 秒杀接口 QPS | ≥ 5000 |
| 秒杀接口响应时间（P99） | ≤ 500ms |
| 普通接口响应时间（P99） | ≤ 200ms |
| 系统可用性 | ≥ 99.9% |
| 支持并发用户数 | ≥ 10000 |

### 4.2 安全需求

| 需求 | 实现方式 |
|------|---------|
| 用户认证 | JWT Token + Spring Security |
| 密码安全 | BCrypt 加密存储 |
| 接口防刷 | 限流 + 隐藏秒杀地址 + 验证码 |
| SQL 注入防护 | MyBatis-Plus 参数化查询 |
| XSS 防护 | 输入过滤 + 输出转义 |
| CSRF 防护 | JWT 无状态认证，天然免疫 |
| 接口权限 | RBAC 角色权限控制 |

### 4.3 可靠性需求

| 需求 | 实现方式 |
|------|---------|
| 数据一致性 | Redis 预扣 + 数据库事务 + 最终一致性 |
| 消息不丢失 | RabbitMQ 消息确认机制（ACK）+ 消息持久化 |
| 接口幂等性 | 秒杀接口基于用户ID+活动ID去重 |
| 优雅降级 | Redis 不可用时降级为数据库直接操作（限制并发） |

### 4.4 可扩展性

- 秒杀服务可独立部署，支持水平扩展
- 数据库支持读写分离
- Redis 支持集群模式
- RabbitMQ 支持集群模式

### 6.2 订单超时取消流程

```
订单服务                    RabbitMQ（延迟队列）                    订单服务（消费者）                    MySQL
   │                              │                                      │                              │
   │─创建订单（待支付）           │                                      │                              │
   │─发送延迟消息（TTL=15min）──→│                                      │                              │
   │                              │──── 15分钟后 ────→                   │                              │
   │                              │─消息转入死信队列──→                  │                              │
   │                              │                                      │─消费消息                     │
   │                              │                                      │─校验订单状态                 │
   │                              │                                      │─更新为"已取消"─────────────→│─UPDATE──→│

   │                              │                                      │─回滚Redis库存               │
```

### 7.3 Redis 数据结构设计

| Key | 类型 | 用途 | 过期时间 |
|-----|------|------|---------|
| `seckill:stock:{activityId}:{goodsId}` | String | 秒杀库存计数 | 活动结束后 1 小时 |
| `seckill:activity:{activityId}` | Hash | 秒杀活动信息缓存 | 10 分钟 |
| `seckill:goods:{goodsId}` | Hash | 商品详情缓存 | 30 分钟 |
| `seckill:limit:{userId}:{activityId}:{goodsId}` | String | 用户是否已参与（幂等标记） | 24 小时 |
| `seckill:path:{userId}:{activityId}:{goodsId}` | String | 动态秒杀地址（MD5） | 活动结束后 1 小时 |
| `seckill:rate:{userId}` | String | 用户访问频率计数 | 1 秒 |
| `user:token:{token}` | String | 用户登录 Token | 30 分钟 |
| `goods:list:{page}:{categoryId}` | String | 商品列表缓存 | 5 分钟 |

## 9. 安全设计

### 9.1 秒杀安全防护策略

```
请求入口
  │
  ├── [CDN / Nginx] ── IP 黑白名单、请求频率限制
  │
  ├── [网关层] ── Token 校验、接口签名验证
  │
  ├── [限流层] ── 令牌桶限流（单用户 QPS 限制）
  │
  ├── [业务层] ── 隐藏秒杀地址、用户限购校验
  │
  ├── [缓存层] ── Redis 原子库存扣减
  │
  └── [异步层] ── RabbitMQ 削峰填谷
```

### 9.2 分布式锁设计

| 场景 | 锁的 Key | 锁的 Value | 过期时间 |
|------|---------|-----------|---------|
| 秒杀库存扣减 | `lock:seckill:{activityId}` | UUID（防止误删） | 10 秒 |
| 订单创建 | `lock:order:{userId}:{activityId}` | UUID | 30 秒 |

### 9.3 数据安全

- 所有敏感数据（密码、手机号）加密存储
- 日志脱敏处理（手机号中间四位用 * 替代）
- 数据库定期备份
- SQL 注入防护（MyBatis-Plus #{} 参数化查询）
